# AGENT_ANTI_REGRESSION_PROTOCOL.md

Последний раз обновлено: 2026-07-16.

Это рабочий контракт для любого агента (или человека-редактора), который правит
репозиторий **THE LEGENDARY POET**. Цель — не «галочка», а реальная защита от
регрессий: этот проект страдает не от падений сборки, а от **тонких ошибок** —
неправильной даты, искажённой строки стиха, дубль-id, который ломает роутинг, или
`biblicalPerspective`, цитирующего текст, которого в стихотворении нет. Такие
ошибки молча проходят `tsc` и `vite build`.

---

## 0. Один источник истины для правил

- Архитектурные/дизайн-правила живут в `ARENA_PROJECT_PROTOCOL.md` — не дублировать.
- Богословско-редакционные — в `src/docs/THEOLOGICAL_GUIDELINES.md`.
- Этот файл — **процесс и защита от регрессий**. Если три файла конфликтуют,
  приоритет у более конкретного, но флагуйте конфликт в PR, не молчите.

## 1. Обязательный пред-коммит гейт (НЕ пропускать)

Перед любым финальным ответом / коммитом выполните по порядку:

```bash
npx tsx scripts/validate-library.ts   # целостность данных: 0 ERROR
npx tsc --noEmit                       # типы: 0 ошибок
npm run build                          # прод-сборка: успех
```

- `validate-library.ts` ловит структурные регрессии (пустые поля, дубли id,
  хронология, рейтинги, отсутствие источников). **Если есть ERROR — чиним, а не
  игнорируем.** WARN допустимы, но каждый WARN должен быть осознанным решением.
- `tsc --noEmit` проверяет ВСЕ файлы, включая спящий `src/components/hall/*`.
  Дормантный код тоже обязан компилироваться — иначе он сгниёт и однажды сломает
  сборку при возвращении в роутинг.
- `npm run build` подтверждает, что ничего не отломано на уровне сборки.

> Эти три команды — дешёвая страховка. Пропускать их ради скорости = вводить
> регрессию knowingly.

## 2. Редакционные правила источников (критично для репутации проекта)

Проект — о реальных людях, реальной истории и реальных стихах. Ошибка в факте
здесь хуже бага в вёрстке.

### 2.1. Иерархия источников (A+, A, B)

- **A+ — первоисточник.** Автограф / академическое Полное собрание сочинений
  (ИМЛИ РАН / «Наука»), ФЭБ (`feb-web.ru`), РВБ (`rvb.ru`), Библиотека Поэта со
  ссылкой на архивный автограф, прижизненные/первые публикации. **Цитату стиха
  всегда сверять по A+.**
- **A — авторитетный вторичный.** Академические Викитека-издания, `culture.ru`,
  именные сайты-музеи (`esenin.ru`, `gumilev.ru`), академические монографии и
  биографии серии ЖЗЛ (Быков, Лекманов/Свердлов, Макеев, Флейшман, Янгфельдт,
  Пигарёв, Тименчик, Богомолов), официальные реабилитационные документы.
- **B — допустимо в крайнем случае.** Качественная журналистика, ссылающаяся на
  архивы (АиФ, «Год литературы»), выверенные образовательные порталы. Только как
  *подтверждение*, никогда как единственное основание для факта.

### 2.2. Правило двух источников

Каждый канонический текст стиха и каждая спорная дата — **минимум по двум
независимым источникам**, хотя бы один из A+. Сверка только по «Википедии» или
одному стихотворному сайту = регресс.

### 2.3. Запрет на фабрикацию

- Не выдумывать стихи, цитаты, даты, «духовные прозрения». Нет источника — нет
  утверждения (или честная пометка «не установлено»).
- Каждое свидетельство (`testimonies`) должно иметь поле `source` (книга/год);
  `sourceUrl` желательно. Валидатор предупредит, если `sourceUrl` нет.
- Если ставите ссылку — **проверьте, что она живая**. Мёртвая ссылка хуже
  отсутствующей. Не угадывайте URL Викитеки/РВБ — откройте и убедитесь.

### 2.4. Класс ошибки, который уже был (учить наизусть)

**Тютчев, «Наш век».** В `text` лежала неканоническая концовка
(`«О Господи! мой боже! верю я! / Помоги моему неверью!»`, «перед глухою
дверью»), а `biblicalPerspective` цитировал совсем другие слова. Реальный
канонический текст (по автографу ИРЛИ Ф.466 №34 и первой публикации в
«Москвитянине» 1851) — `«Впусти меня! — Я верю, Боже мой! / Приди на помощь
моему неверью!..»`, «перед замкнутой дверью». **Вывод:** поле, комментирующее
стих, обязано согласовываться с самим `text`. Эта сверка делается глазами —
автомат надёжно её не делает (легитимно цитируется и Писание, и стих), поэтому
она в ручном чек-листе ниже.

## 3. Чек-лист перед правкой стиха / разбора

- [ ] `text` сверен по 2+ источникам, хотя бы один A+ (ФЭБ/РВБ/Викитека/Библиотека Поэта).
- [ ] `biblicalPerspective` / `analysis` цитируют **те же слова**, что лежат в `text` (или явно помечены как парафраз/цитата Писания со ссылкой).
- [ ] `year` попадает в жизненный диапазон поэта (валлидатор предупредит, если нет).
- [ ] Библейская ссылка (книга/глава/стих) точна и проверена по тексту Писания.
- [ ] Я не сделал поэта «верующим» без серьёзных оснований (см. THEOLOGICAL_GUIDELINES).

## 4. Кодовые правила (кратко, дубликаты — в ARENA_PROJECT_PROTOCOL.md)

- **Данные модульны:** новый поэт — отдельный файл `src/data/library/<name>.ts`,
  экспорт через `src/data/library/index.ts`. Не собирать обратно один гигантский
  `poets.ts`.
- **Никаких emoji в UI** — только Lucide/custom SVG.
- **Движок эссе — данные, не вёрстка:** новые блоки добавляются через
  расширение `EssayBlock` + компонент в `blocks.tsx` + `case` в switch. Guard
  `never` делает пропуск нового типа ошибкой компиляции — не обходите его.
- **Без `dangerouslySetInnerHTML`** для пользовательского/редакторского текста;
  используется безопасный `withGold` (только `**...**`).
- **localStorage-записи — через `safeWrite`** (не ронять клик-хендлер в приватном
  режиме Safari / при превышении квоты).
- **Тяжёлый 3D (three.js) изолирован:** `src/components/hall/*` сейчас дормантен
  и **не должен** попадать в бандл. Если возвращаете зал в роутинг — через
  `React.lazy` + `Suspense` + ErrorBoundary + DOM-фолбэк (см. `HallOrFallback`).

## 5. Правила совместной работы агентов (важно!)

Над репозиторием могут параллельно работать несколько агентов с близкими
задачами. Чтобы не топить друг друга:

1. **Предпочитайте аддитивные изменения.** Новые файлы (`scripts/*`,
   `docs/*`, новый поэт) почти не конфликтуют. Правки общих файлов
   (`poets.ts`, `index.ts`, `RESEARCH_SOURCES.md`) — зона риска.
2. **Не переписывайте чужие свершения вслух.** Если видите правку, которой не
   было в вашем базовом коммите, — не откатывайте её автоматически; разбирайтесь.
3. **Каждый коммит — атомарный и подписанный.** В сообщении: ЧТО изменено и
   ПОЧЕМУ, со ссылкой на источник для фактических правок.
4. **Перед push — `git pull --rebase`** и повтор прогнать гейт из §1.
5. **Не редактируйте `package.json` руками в Arena** (правило проекта). Удаление
   неиспользуемых зависимостей (`vite-plugin-singlefile`, `@studio-freight/lenis`)
   — вне Arena или approved-командой.

## 6. Что НЕ делать (антипаттерны, замеченные в дикой природе)

- ❌ Сглаживать «ради красивости» хронологию или спорные факты (дуэль Лермонтова,
  гибель Есенина). Держите версию большинства источников и честно помечайте
  альтернативные.
- ❌ Превращать поэта в святого или в злодея. Талант ≠ спасение; грех ≠
  обесценивание дара (см. THEOLOGICAL_GUIDELINES).
- ❌ Оставлять `console.log` / закомментированный мусор в проде.
- ❌ Чинить WARN валидатора «заглушкой» (например, пустым `sourceUrl`). Лечите
  причину.
- ❌ Считать сборку зелёной = контент верный. Сборка не знает, правильный ли год
  у Пушкина. Для контента — §2 и §3.
